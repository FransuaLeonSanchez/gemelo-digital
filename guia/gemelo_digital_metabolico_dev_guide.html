¿Qué estás construyendo?

Lee esto primero antes de tocar una sola línea de código.

💡

Concepto central: Un gemelo digital metabólico es una réplica computacional en tiempo real del estado metabólico de una persona. Recibe datos continuos de sensores físicos (glucosa, actividad, sueño, estrés), los procesa con modelos de IA y produce predicciones de riesgo + recomendaciones personalizadas. El "gemelo" aprende y se actualiza constantemente para volverse más preciso con cada usuario.

🎯 Objetivo del sistema

Detectar el riesgo de síndrome metabólico en etapas tempranas en adultos de 20-45 años, antes de que aparezcan síntomas clínicos. El sistema debe ser capaz de:

# LO QUE EL SISTEMA DEBE HACER 1. Recibir datos continuos de CGM (glucosa), wearable (HRV, pasos, sueño) y app (hábitos) 2. Construir un modelo metabólico individual por usuario (gemelo digital) 3. Predecir respuesta glucémica a comidas y hábitos ANTES de que ocurran 4. Calcular un "Índice de Carga Metabólica" (ICM) en tiempo real: score 0-100 5. Generar recomendaciones personalizadas diarias basadas en el perfil del usuario 6. Alertar cuando el ICM supere umbrales de riesgo definidos

Flujo completo de datos

Cómo viajan los datos desde el cuerpo del usuario hasta la recomendación final.

1

📡 Captura de datos — Sensores IoT

El usuario lleva 3 fuentes de datos activas simultáneamente:

CGM → glucosa cada 5 min Wearable → HRV, pasos, BPM, sueño App → comidas, estrés, síntomas
Protocolo de conexión: Bluetooth Low Energy (BLE) para wearable y CGM → App móvil → API via HTTPS. Los datos del CGM llegan cada 5 minutos de forma automática. Los del wearable, cada 1 minuto. Los de la app, cuando el usuario los registra (mínimo 3 veces al día: desayuno, almuerzo, cena).
2

☁️ Ingesta y validación — Backend API

El backend recibe, valida y normaliza todos los datos entrantes.

Validar rangos fisiológicos Detectar datos faltantes Timestamping en UTC
Reglas de validación críticas:
• Glucosa: rango válido 40–400 mg/dL. Fuera de rango = alerta inmediata.
• HRV: valores negativos o >200ms = descartar, marcar como error de sensor.
• Si hay gap de datos mayor a 15 minutos en CGM, notificar al usuario.
• Todos los timestamps deben convertirse a UTC antes de almacenar.
3

🗄️ Almacenamiento — Base de datos

Los datos se almacenan en dos bases complementarias.

PostgreSQL → datos estructurados TimescaleDB → series de tiempo
Importante: Los datos de glucosa y wearable son series de tiempo densas. Usar TimescaleDB (extensión de PostgreSQL optimizada para series de tiempo). Los datos de hábitos y perfiles de usuario van en PostgreSQL estándar. Retención de datos: mínimo 2 años por usuario.
4

🧠 Procesamiento IA — Motor del Gemelo Digital

El núcleo del sistema. Corre en un microservicio independiente.

LSTM → predicción glucémica Random Forest → clasificación riesgo Índice de Carga Metabólica (ICM)
Cuándo se ejecuta: El motor IA corre en 3 momentos: (1) cada 15 minutos de forma automática para actualizar el ICM, (2) cada vez que el usuario registra una comida para predecir la respuesta glucémica, (3) una vez al día a las 7am para generar el resumen y recomendaciones del día.
5

📲 Entrega al usuario — App y Notificaciones

El resultado final que ve el usuario en su dispositivo.

ICM en tiempo real (0–100) Alertas push críticas Recomendaciones diarias Gráfico de glucosa
Tipos de notificación:
• Alerta crítica: glucosa <70 o >180 mg/dL → push inmediato.
• Alerta de riesgo: ICM supera 70 → notificación + recomendación urgente.
• Resumen diario: 7am → resumen del día anterior + plan del día.

Módulos del sistema

Haz clic en cada módulo para ver los detalles de implementación.

🔐
Autenticación
auth_service
Registro, login, gestión de sesiones y permisos.
  • JWT tokens con refresh token (24h acceso / 30d refresh)
  • Registro con email + verificación
  • Roles: paciente, médico, admin
  • OAuth2 (Google) para registro rápido
  • Encriptación de datos sensibles con AES-256
📡
Integración Sensores
sensor_gateway
Conexión BLE con CGM y wearables compatibles.
  • Soporte CGM: Dexcom G6/G7 API, Libre 3 API
  • Soporte Wearable: Fitbit API, Garmin Connect IQ, Apple HealthKit, Google Health
  • Protocolo BLE para conexión directa desde app
  • Fallback: entrada manual de glucosa si no hay CGM
  • Sync automático en background cada 5 min
📝
Registro de Hábitos
habits_service
Módulo de autoreporte de comidas, sueño y estrés.
  • Registro de comidas: búsqueda en base de datos nutricional peruana (CENAN)
  • Carga glucémica calculada automáticamente por porción
  • Registro de sueño: hora acostarse, hora levantarse, calidad 1-5
  • Estrés: escala visual 1-10 + notas libres
  • Recordatorios configurables (push notifications)
🧬
Gemelo Digital Core
digital_twin_engine
El corazón del sistema. Modelo individual por usuario.
  • Un objeto "Twin" por usuario almacenado como modelo serializado
  • Reentrenamiento automático cada 7 días con datos nuevos
  • Modelo base: preentrenado con dataset público (NHANES + ENDES Perú)
  • Fine-tuning individual: primeras 2 semanas son de calibración
  • Versionado de modelos: guardar historial de versiones del gemelo
📊
Índice de Carga Metabólica
icm_calculator
Score 0-100 que resume el riesgo metabólico en tiempo real.
  • ICM = promedio ponderado de 5 sub-índices
  • Sub-índice glucémico (peso: 35%) — basado en CGM
  • Sub-índice de actividad (peso: 20%) — basado en pasos y actividad
  • Sub-índice de sueño (peso: 20%) — basado en horas y calidad
  • Sub-índice de estrés (peso: 15%) — basado en HRV
  • Sub-índice nutricional (peso: 10%) — basado en carga glucémica
  • Umbrales: 0-39 verde / 40-69 amarillo / 70-100 rojo
💬
Motor de Recomendaciones
recommendation_engine
Genera recomendaciones personalizadas basadas en el gemelo.
  • Banco de 200+ recomendaciones parametrizadas por variable
  • Selección basada en el delta negativo más alto del ICM del usuario
  • Nunca repetir la misma recomendación 2 días seguidos
  • Tono: amigable, en español peruano, sin tecnicismos médicos
  • A/B testing: registrar qué recomendaciones generan más adherencia

Motor de Inteligencia Artificial

Detalle técnico del núcleo del gemelo digital.

🔮 Modelo 1 — Predicción Glucémica (LSTM)

Predice cómo va a cambiar la glucosa del usuario en las próximas 2 horas.

# ARQUITECTURA LSTM input_features = [ 'glucose_last_12h', # Serie de tiempo de glucosa (144 lecturas) 'meal_glycemic_load', # Carga glucémica de la última comida 'minutes_since_meal', # Tiempo desde última comida 'steps_last_hour', # Actividad física reciente 'hrv_current', # HRV actual (proxy de estrés) 'hour_of_day', # Hora del día (patrón circadiano) ] output = 'glucose_next_30_60_120_min' # Predicción a 3 horizontes architecture: LSTM(128) → Dropout(0.2) → LSTM(64) → Dense(3) loss: MAE (Mean Absolute Error) target_error: < 15 mg/dL (clínicamente aceptable)

🎯 Modelo 2 — Clasificación de Riesgo (Random Forest)

Clasifica el nivel de riesgo metabólico actual: bajo / medio / alto.

# FEATURES DEL RANDOM FOREST features = [ 'glucose_cv_pct', # Coeficiente de variación glucémica (últimos 7 días) 'time_in_range_pct', # % tiempo en rango 70-140 mg/dL 'avg_postprandial_peak', # Pico promedio postprandial 'avg_sleep_hours', # Promedio horas de sueño (7 días) 'avg_daily_steps', # Promedio pasos diarios (7 días) 'avg_hrv', # HRV promedio (indicador de estrés) 'tyg_index', # Índice TyG = ln(TG/2 × glucosa/2) si hay labs 'waist_cm', # Circunferencia abdominal ] output_classes: ['bajo', 'moderado', 'alto'] n_estimators: 200 target_accuracy: > 85%

📐 Índice TyG — Proxy de Resistencia a la Insulina

Cuando el usuario tiene datos de laboratorio, calcular este índice como indicador indirecto de resistencia a la insulina (RI).

# FÓRMULA DEL ÍNDICE TyG def calculate_tyg(triglycerides_mgdl, fasting_glucose_mgdl): # Ambos valores en mg/dL, obtenidos de examen de sangre tyg = ln(triglycerides_mgdl / 2) + ln(fasting_glucose_mgdl / 2) return tyg # INTERPRETACIÓN # TyG < 8.5 → Sin resistencia a la insulina # TyG 8.5–9.0 → Riesgo moderado de RI # TyG > 9.0 → Alta probabilidad de resistencia a la insulina
